# Drop-in starter for non-Claude agents

Paste the prompt below verbatim into any agent (Cursor, ChatGPT, Llama, custom harness, etc.) to get it to participate in `agent-show-and-tell` without needing the SKILL.md frontmatter machinery. Replace the bracketed `[...]` values with the actual identity/topic before pasting.

The starter has two parts. Use **Part A (Reporter)** for every participating agent. Use **Part B (Reader)** for the *one* agent (or human) doing the roundup at the end.

---

## Part A — Reporter prompt

```
You're participating in a multi-agent "show-and-tell" — a fire-and-forget standup
where each agent in a fleet writes one short report describing what it knows and
what it's working on. There is no grilling, no consensus, and no merging.

YOUR JOB: write exactly one markdown file to:

  [output-dir]/[your-agent-name].md

The default [output-dir] is `.show-and-tell/[topic-slug]/` relative to the
shared repo root. The human running this can override it with an absolute path
if you don't have a stable shared CWD (e.g. you're an automation job writing
to /var/run/...). Use whatever path the human provides; only fall back to the
default if no path was given.

Write the file before doing anything else. Use this exact template (the YAML frontmatter is
mandatory and must come first; do not add or remove sections; write "(none)" if
a section truly does not apply):

---
agent: [your-agent-name]
topic: [topic-slug]
session_started: [ISO8601 timestamp now]
facets:
  project: [repo or product slug, or "none"]
  harness: [your runtime — e.g. cursor, claude-code-cli, openhands, custom-sdk]
  provider: [model API behind you — e.g. anthropic, openai, google, local-ollama]
  surface: [where the human interacts — terminal, web, vscode, slack-bot]
  machine: [hostname or stable id, or "unknown"]
  user: [username at the keyboard, or "automation"]
---

# [your-agent-name] — [topic-slug]

## What I'm working on
(one paragraph — the active task, not the project at large)

## What I know
- **<short claim>** — source: <file:line, command output, prior decision, or "inferred">; confidence: high | medium | low
- (more bullets...)

## Open questions
- (things you're uncertain about that another agent might have already resolved)

## What other agents should know
- (1–5 most useful surfaceable items — not a re-listing of "What I know")

## Last action
(one line: most recent concrete thing you did, with timestamp)

RULES:
1. Do NOT read other agents' reports in [output-dir] before writing yours.
   Blind reports prevent groupthink and let the reader detect real consensus
   instead of echoes.
2. Confidence levels: "high" requires a checkable source you can cite (file
   path, command output, signed-off doc) — session memory does NOT qualify.
   "medium" = inferred from possibly-stale signal. "low" = guess. If you
   can't justify "high," don't claim it.
3. Keep the report under ~60 lines. If it's longer, the topic is too broad —
   split it.
4. If a file with your agent name already exists in [output-dir], handle the
   identity collision in this order:
   (a) If you have a human in the loop, STOP and surface the collision.
   (b) If you're an automation job with no human, append a UTC ISO8601
       timestamp suffix to your filename: [your-agent-name]-[ISO8601].md.
       Then write a one-line marker file _collision-[your-agent-name].md
       noting the collision so the reader can detect it.
   Never overwrite an existing report.
5. Once your file is written, you're done with THIS task. "Don't loop" means
   don't loop within this task — your outer cron or scheduler is unaffected.
6. If your harness has no filesystem write capability, you cannot participate
   directly. Tell the human; they'll need to relay your report.
7. If your reports could contain secrets, ARNs, internal hostnames, or PII,
   confirm with the human whether the [output-dir] is appropriate audience
   before writing — show-and-tell directories are typically committed to git.
```

## Part B — Reader prompt (only one agent runs this)

```
You're the designated reader for a multi-agent "show-and-tell." Every other
agent in the fleet has written one report file under:

  .show-and-tell/[topic-slug]/<agent>.md

YOUR JOB: read every report (not yours; if you're also a reporter, read yours
last to avoid biasing the collation), then write a single roundup to:

  .show-and-tell/[topic-slug]/roundup.md

Use this exact structure:

# [topic-slug] — roundup

**Generated by:** <your-agent-name> at <ISO8601>
**Participants reported:** <list of agent names whose files exist>
**Missing:** <participants from _meta.yaml who didn't submit, if _meta.yaml exists>

## Fleet shape

| Facet | Values present |
|---|---|
| project | <values from frontmatter, with counts> |
| harness | ... |
| provider | ... |
| (etc.) | ... |

## Consensus
(facts that appeared in 2+ reports without contradiction; cite each agent and source)
- **<claim>** — agent-A (`<source>`), agent-B (`<source>`)

## Solo facts
(facts only one agent reported; group by agent)
### From <agent-A>
- ...

## Conflicts
(facts where two reports disagree — name them, do NOT resolve them)
- **<topic>** — agent-A says X (source: ...); agent-B says Y (source: ...)

## Cross-facet observations
(things visible only when slicing reports by facets — e.g. "all three cursor agents reported the same MCP timeout"; cite supporting agents)
- ...

## Open questions across the fleet
(merged from each agent's "Open questions" section; tag with which agent asked)
- ...

## Suggested next actions
(short, specific; cite the report(s) that motivate each)
- ...

RULES:
1. Do NOT invent claims. Every line traces to a source report.
2. Do NOT resolve conflicts. Naming them is the job; resolving is a different
   protocol (peer-grill).
3. If a participant in _meta.yaml didn't submit a report, list them as missing.
   Do not proceed as if they had nothing to add.
4. Pause and re-read each report once before collating. Roundups written too
   fast tend to drop the conflicts.
```

---

## Notes for the human running this

- Pick a `topic-slug` first (e.g. `monday-status-2026-05-01`). Tell every agent the same slug.
- Pick distinct `agent-name`s for each agent (e.g. `claude-laptop`, `cursor-mac`, `local-llama-vm`). Identity collisions are detected but they cost a turn.
- The shared directory must be reachable from every agent. If your agents are on different machines, sync it via git, a shared drive, or by relaying through yourself.
- Commit `.show-and-tell/<topic>/` to git when done so the snapshot is auditable.
